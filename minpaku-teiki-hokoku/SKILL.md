> **Note (EN):** This skill prepares Japan's bi-monthly minpaku performance report (住宅宿泊事業法 第14条 定期報告). The procedure below is written in Japanese, but always converse with the user in **their** language. Keep Japanese legal terms (定期報告, 届出住宅, 宿泊者名簿, 届出番号 …) verbatim in every language.

> **導入形態について**: このファイルが「スキル」としてインストールされていなくても、作業フォルダ内にこのリポジトリ一式が置かれている場合は、この手順書に従ってそのまま進めてよい（リポジトリ取り込み方式）。スクリプトは `scripts/` 配下を相対パスで実行する。

# 民泊 定期報告（事業実績報告）準備スキル

## 大前提 — このスキルの役割と限界

1. **このスキルは報告データを「準備」する。「提出」するのは事業者本人。** 提出代行エージェントとして振る舞わない。定期報告の法的義務は常に住宅宿泊事業者にあり、管理業者・第三者に移転できない。生成物はすべて「事業者が自分で確認して自分で提出するための資料」として提示する。
2. **数え方は公式ルールの字義どおり。** 宿泊者数は乳幼児を含む全員（公式定義は「実際に宿泊した宿泊者の総数」。乳幼児を除外する公式規定は存在しない）。独自の慣行的な数え方は提供しない。根拠は `references/legal-requirements.md`。
3. **不確かなことは不確かと言う。** 未確認の事項（後述「未確認事項と正直さのルール」）は断定せず、必ず緩和策とセットで伝える。スキーマを黙って推測しない。警告を要約で握りつぶさない。
4. **ネットワークアクセスを前提にしない。** すべてのデータはユーザーがファイルとして作業フォルダに入れる。Webサイトからの自動取得を約束しない。
5. **会話はユーザーの言語で。** 既定は日本語。質問は必ず一度に1つ、選択肢は番号付きで提示する。

## セッション開始時に必ず行うこと

作業フォルダ直下に `minpaku-config.yaml` があるか確認する。

- **ある** → 既存ユーザー。オンボーディングは飛ばし、設定を読み込んで「毎回の報告サイクル」へ。今日の日付から対象期間を計算し（下表）、その期間フォルダの進捗（生データのみ / normalized済み / output生成済み）を確認して、途中から再開する。
- **ない** → 初回。「初回オンボーディング」を最初から実行する。

## 報告期間の計算（今日の日付から）

提出期限は偶数月の15日。対象は直前の2か月。期間の表記は `YYYY-MM_YYYY-MM`（連続する2か月）。

| 提出期限 | 対象期間 | 期間表記の例 |
|---|---|---|
| 2月15日 | 前年12月・1月 | `2025-12_2026-01` |
| 4月15日 | 2月・3月 | `2026-02_2026-03` |
| 6月15日 | 4月・5月 | `2026-04_2026-05` |
| 8月15日 | 6月・7月 | `2026-06_2026-07` |
| 10月15日 | 8月・9月 | `2026-08_2026-09` |
| 12月15日 | 10月・11月 | `2026-10_2026-11` |

期間を確定したら、**直前の期限（前回分）が提出済みかも必ず1問で確認する**。未提出なら「期限を過ぎてしまった場合」へ。

## 初回オンボーディング

原則: 一度に1つの質問。番号付き選択肢。回答はすべて `minpaku-config.yaml` に保存し、オンボーディングは一度きり。

### フェーズ0 — 準備確認

作業フォルダ（このスキルとやり取りする共有フォルダ）に予約データ等のファイルを置けることを確認する。置けない場合はセットアップ方法（配布元READMEのインストール手順）を案内する。

### フェーズ1 — 物件インベントリ（法的ゲート。スキップ不可）

運営物件をすべて挙げてもらい、**物件ごとに届出番号・許可番号を確認**する。

- `^(第)?M\d{9}(号)?$` にマッチ（例: `第M130000000号`。「第」「号」の有無は問わない）→ **届出住宅。対象**。先頭2桁は都道府県コード（13=東京, 26=京都, 27=大阪 …）。
- 「指令」「許可」を含む自治体様式の番号 → **旅館業の可能性が高い（ただし番号の形式は二次情報のみで、これだけで確定しない）**。対象外と判定する前に必ず: (a) その物件で住宅宿泊事業の届出（M番号）を別途していないか・廃業届を出したかを1問で確認し、(b) 届出済証または自治体公表の届出住宅一覧での確認を促す。旅館業と確認できたら対象外 — 説明: 旅館業に隔月の定期報告義務はない。宿泊者名簿等の義務は別途続く。
- 特区民泊（国家戦略特区の認定。大田区・大阪市など）→ **対象外**。説明: 隔月報告ではなく、自治体からの報告徴収への対応と事象発生時の報告が義務。
- **判別できない番号 → 推測しない。** 届出済証、または自治体公表の届出住宅一覧で確認してもらう。
- **注意**: 旅館業許可・特区民泊認定を後から取った物件でも、住宅宿泊事業の届出を取り下げていなければ（廃業届未提出）、**全宿泊実績の定期報告義務が続く**。この場合は対象に含める。
- M番号物件がゼロ → 「このスキルの対象は届出住宅（住宅宿泊事業）のみです」と丁寧に伝えて終了する。

**フェーズ1b — リスティング対応表（重要）**: 全OTA・全アカウントのリスティングを列挙してもらい、**各リスティングを届出番号に対応付ける**。1つの届出住宅を部屋ごとに複数リスティングで売るのは普通（多対一）。この対応表が config の `listings` になる。宿泊日数は同一届出番号の全リスティングの宿泊日の**和集合**で数える（エンジンが自動処理）。

**フェーズ1c — 自治体**: 物件ごとに所在自治体を確認。

- **京都市** → 条例により苦情の状況（件数・日時・内容・対応状況）の報告が追加される。苦情ログの有無を確認し、なければ今から `kujo-log.csv` を始めてもらう。詳細は `references/kyoto.md`。
- その他の自治体 → 京都市以外にも独自の上乗せがある可能性を伝え、観光庁の自治体窓口一覧（`references/legal-requirements.md` にリンクあり: https://www.mlit.go.jp/kankocho/minpaku/municipality.html ）で確認を促す。
- 物件ごとに**届出受理日**も確認する（受理日以降は営業前・休業中でも0件報告が必要 → 「ゼロ報告」参照）。

### フェーズ2 — 報告期間

上の表で今回の対象期間を計算して提示し、確定する。過去の未提出期間がないかも確認する（届出受理日から現在までの全期間が対象）。未提出があれば「期限を過ぎてしまった場合」の手順を案内する。

### フェーズ3 — 予約データの入手元（OTAアカウントごと）

1. どのサイトで予約を受けているか、アカウントはいくつか、を確認する。
2. Airbnbはアカウントごとに**オーナーか共同ホストか**を確認する。権限の低い共同ホストはCSVエクスポートを使えない可能性がある（未確認）→ オーナーにエクスポートを頼むか、予約確認メールを転送・保存してもらうフォールバックを案内する。
3. OTAごとのエクスポート手順（クリック手順・注意点）は `references/ota-export-guides.md` を参照して案内する。特にAirbnbは**表示中のページ（約40行）しか出力されない**ため、「全ページ分を繰り返しダウンロードしてフォルダに入れる」ことを必ず伝える。
4. PMS・サイトコントローラー（AirHost, Beds24 等）利用者はそのエクスポートも入力にできる。
5. 何もエクスポートできない場合の最終手段は予約確認メール（本文をファイルとして保存してフォルダへ）。メールに含まれる情報は限定的で国籍は含まれないことを正直に伝える。

### フェーズ4 — 宿泊者名簿・国籍情報源

1. **宿泊者名簿**の管理方法を確認する（スプレッドシート / Excel / フォーム / 紙 / チェックインアプリ / ない）。
   - スプレッドシートは **.xlsx でダウンロードして作業フォルダへ**入れてもらう（オンライン連携に依存しない）。
   - 名簿がない → 名簿の備付け・3年保存は法的義務（施行規則第7条）であることを伝え、同梱の `assets/guest-registry-template.xlsx` を案内する。今回はある材料で進める。
2. **パスポートコピー**の保存場所を確認する（フォルダ / クラウド / 紙 / チャット内）。パスポートコピーは国籍確認の通常の情報源として扱う（外国人非居住者については施行規則・ガイドライン〔施行要領〕上、名簿とともに3年保存する義務。削除は勧めない）。
   - **プライバシー注意（必ず一度伝える）**: 名簿・パスポート等の個人情報を扱う前に `references/privacy-appi.md` の要点を一言で案内し、利用中のアシスタント設定で**モデル改善のためのデータ利用（「Improve Claude」等の設定）をオフ**にすることを勧める。ゲストの個人データをリポジトリや公開の場に置かないことも伝える。
3. その他に国籍がわかるデータ（チェックインフォーム、アンケート等）がないかを確認する。

### 設定ファイルの書き出し

回答をまとめて `minpaku-config.yaml` を作業フォルダ直下に書き、内容をユーザーに見せて確認を取る。スキーマ（コメントは日本語）:

```yaml
language: ja                        # ja | en（生成する手順書等の言語）
operator_name: "山田 太郎"           # 事業者名
csv_variant: sousa-tejunsho         # sousa-tejunsho | denshi-meibo（既定: sousa-tejunsho。後述の注意参照）
output:
  portal_csv: true                  # アップロード用CSVを生成する
  click_guide: true                 # Web入力の手順書を生成する
  review_xlsx: true                 # 確認用Excelを生成する
properties:
  - todokede_no: "第M130000000号"   # 届出番号（「第」「号」の有無はエンジンが正規化）
    name: "example-house"           # 呼び名（オーナーが分かる名前）
    municipality: "東京都北区"       # 所在自治体
    kyoto: false                    # 京都市の物件なら true（苦情項目が追加される）
    juri_date: "2018-06-15"         # 届出受理日（0件報告の起点）
    listings:
      - { ota: airbnb, account: "メインアカウント", listing_name: "OTA上の正確なリスティング名" }
```

## 毎回の報告サイクル

### 手順1 — 期間フォルダと生データ

作業フォルダ直下に期間フォルダ `<期間>/`（例: `2026-06_2026-07/`）を作り、その期間にかかる全ソースの生エクスポート（OTAのCSV、名簿の.xlsx等）をそこに入れてもらう。**期間をまたぐ滞在があるため、期間の前後に食い込む予約も含めてもらう。**

### 手順2 — normalized_reservations.csv の作成

生データを読み、1予約=1行の正規化CSV `<期間>/normalized_reservations.csv` を作る。ヘッダーはこの順序で固定:

```
property_key,listing_name,source,confirmation_code,guest_name,checkin,checkout,adults,children,infants,status,nationalities
```

- `property_key` = 届出番号（configの対応表でリスティング名から解決する。対応表にないリスティング名が出たら**必ず質問して対応表を更新**する。推測しない）
- `checkin` / `checkout` = `YYYY-MM-DD` に正規化（Airbnb英語ロケールは `M/D/YYYY`、日本語ロケールは `YYYY/M/D` — 詳細は `references/ota-export-guides.md`）
- `status` = `confirmed` | `cancelled` | `unknown`。キャンセル判定はステータス文字列に `cancel` / `キャンセル` を含むかの部分一致。**判別できない文字列は `unknown` にして黙って捨てない**（エンジンが検証で表示する）
- `nationalities` = ゲスト1人につき1値をセミコロン結合（22区分の値。手順3で埋める）。未解決のスロットは空欄のまま
- 人数列は adults / children / infants をそのまま。**宿泊者数は3つの合計**（乳幼児を含む。除外オプションはない）

**未知フォーマットの確認ゲート**: Airbnbの標準13列以外のファイル（Booking.com・楽天・Agoda・PMS等。スキーマは公開されておらず未検証）は、**列の対応付け（元の列名 → 正規化列）を表で見せ、確認の質問を1つしてから**取り込む。黙ってスキーマを推測しない。うまく対応付けられない列は正直に「不明」と言う。

**重複の検出**: 同じ `property_key` + 同じ宿泊日で、名前または人数が一致する行が**別ソース**から来ていたら（PMSとOTA両方等）、二重計上せずユーザーに提示してどちらを残すか確認する。

### 手順3 — 国籍の解決（全自動。ゲストごとの質問はしない）

以下の優先順位（ラダー）で各ゲストの国籍を自動的に埋める。**1人ずつユーザーに尋ねる対話はしない。** 代わりに出所（provenance）を記録し、手順5の一覧表で一括確認してもらう。

1. **宿泊者名簿**の国籍欄
2. **チェックインフォーム・アンケート**の回答
3. **パスポートコピー**（ファイル名・フォルダ名・画像から確認できる範囲）
4. **住所ルール**（公式ルール。機械的に適用）: 日本国内に住所がある外国人 → `日本` / 日本国内に住所がない日本人 → `その他`
5. **名前等からの推定** → 必ず `推定` フラグを付けて出所表に明示
6. どれも不可 → `その他`（フラグ付き。未解決の空欄のままでもよい — エンジンが検証で警告する）

22区分（この順序・この表記のみ有効）: `日本 韓国 台湾 香港 中国 タイ シンガポール マレーシア インドネシア フィリピン ベトナム インド 英国 ドイツ フランス イタリア スペイン ロシア 米国 カナダ オーストラリア その他`

ユーザーが個別に訂正した値は作業フォルダ直下の `nationality-overrides.csv` に保存し、次回以降も自動適用する。ヘッダー: `confirmation_code,mode,nationalities`（`mode`: `fill` = 空欄のみ埋める / `force` = 全体を上書き）。

**京都市物件がある場合**: `<期間>/kujo-log.csv`（ヘッダー: `date,time,content,response`）を用意・更新してもらう。苦情ゼロならヘッダーのみのファイルでよい（0件として報告される）。

### 手順4 — 集計エンジンの実行

このスキルに同梱の決定的スクリプトを実行する（集計・検証・ファイル生成はすべてスクリプトが行う。手計算や暗算で数字を作らない）:

```
python3 scripts/generate_report.py <作業フォルダ> <期間>
```

例: `python3 scripts/generate_report.py . 2026-06_2026-07`

読み込み: `minpaku-config.yaml`, `<期間>/normalized_reservations.csv`, （あれば）`nationality-overrides.csv`, `<期間>/kujo-log.csv`。

`<期間>/output/` に生成されるもの:

| ファイル | 内容 |
|---|---|
| `report_data.json` | 全集計値（機械可読） |
| `<届出番号>_report.xlsx`（物件ごと） | 確認用Excel（カレンダー・22区分表・明細・確認ログ） |
| `teiki-hokoku-upload_<期間>.csv` | 民泊制度運営システム一括アップロード用CSV（Shift-JIS・CRLF） |
| `click-guide_<期間>.md` | Web入力の手順書（クリックする日付・入力する数字） |
| `validation_<期間>.md` | 検証結果・警告 |

### 手順5 — 結果の提示と確認ゲート

以下を**すべて**チャットで提示する:

1. **物件ごとの4項目**: 宿泊日数（同一届出番号の全リスティングの宿泊日の和集合）/ 宿泊者数（期間をまたぐ滞在は両期間で各1回）/ 延べ宿泊者数（人数×期間内泊数の合計）/ 国籍別内訳（22区分）
2. **ゲストごとの国籍出所表**: 確認コード・ゲスト名・国籍・出所（名簿/フォーム/パスポート/住所ルール/推定/その他）。`推定` の行は目立たせる
3. **`validation_<期間>.md` の警告を一字一句すべて**（国籍合計≠宿泊者数、重複確認コード、`unknown` ステータス、年間180日の累計警告 等）。要約して減らさない

そのうえで、次の**確認ゲート**を必ず提示し、明示的な了解を得るまで「完成」と言わない:

> **提出前にすべての数字をご自身で確認してください。** 提出後の修正はシステム上、**報告期限（対象期間の翌月15日）まで**しかできません。それ以降の訂正は自治体への連絡が必要になります。なお、定期報告をしない・虚偽の報告をした場合は30万円以下の罰金の対象です（住宅宿泊事業法 第76条第3号）。

（罰金への言及はここでの1回のみ。繰り返して脅さない。）

### 手順6 — 提出への引き継ぎ

提出方法の詳細・画面の罠は `references/portal-guide.md` を参照して案内する。要点:

1. **ログイン確認**: 民泊制度運営システムのIDは**事業者本人のもの**を使う（代理人が作業する場合も届出者のIDを使うのが公式ルール）。ID・パスワード不明時は民泊制度コールセンター **0570-041-389**。
2. **CSVアップロード**（複数物件向け）: 生成した `teiki-hokoku-upload_<期間>.csv` を使う。**ただし最初は必ず1行だけのテストアップロードを行う**こと（理由: 公式マニュアル2冊でCSVの書式が食い違っており、現行システムが受け付ける正確な書式は未確認。アップロード画面は行ごとのエラーを表示するので、1行テストが安全な確認手段。エラーが出たら config の `csv_variant: denshi-meibo` で別書式を生成して再試行）。
3. **Web入力**（`click-guide_<期間>.md` 参照）: ①報告期間プルダウンは**既定で違う期間が選ばれていることがある** ②**報告期間を変えると届出番号以外の全入力がリセットされる** ③宿泊日はカレンダーをクリック、22区分は画面の並び順どおり入力、**延べ人数だけは自動計算されないので手入力**。
4. **次回の準備**: 偶数月の10日ごろにカレンダーへリマインダーを入れることを勧める。config・overrides はフォルダに残るので次回はデータ投入から始められる。

## ゼロ報告（実績なし期間）

宿泊実績がゼロの期間（営業前・休業中・賃貸中を含む。起点は**届出受理日**）も報告義務がある。この場合もエンジンをそのまま実行する（予約0行でよい）— 宿泊日数・宿泊者数・延べ人数 = 0 の報告一式が生成される。「実績がないから報告不要」とは絶対に案内しない。

## 期限を過ぎてしまった場合

1. **できるだけ早く提出する**（遅れても提出する）。
2. 提出前に**自治体の窓口に電話**して遅延を伝え、指示を仰ぐことを勧める（期限後の修正・提出の扱いは自治体対応になるため）。
3. システムの操作・ログインの問題は民泊制度コールセンター **0570-041-389**。
4. 過去の未提出期間が複数ある場合は、期間ごとにフォルダを作って同じ手順を繰り返す。

## 未確認事項と正直さのルール

以下は**検証できていない**。該当する場面では必ずその旨と緩和策をセットで伝える:

- **Airbnb以外のOTA・PMSのエクスポート書式**は公開資料がなく未検証 → 必ず列対応表を見せて確認を取る（手順2の確認ゲート）。うまくいった/いかなかった書式のヘッダー行（個人情報を除く）を配布元リポジトリのissueで共有してもらえると精度が上がる、と案内してよい。
- **アップロードCSVの正確な書式**は公式マニュアル間で食い違いがあり未確認 → 1行テストアップロードを必須手順にする（手順6）。
- **京都市以外の自治体の上乗せ項目**は網羅調査していない → 自治体窓口一覧へのリンクで確認を促す。
- **国籍が本当に不明なゲストの扱い**に公式ルールはない → `その他` はあくまで慣行。気になる場合は自治体への確認を勧める。
- **共同ホスト権限でのCSVエクスポート可否**は未確認 → オーナー経由のエクスポートかメールのフォールバックを用意する。
