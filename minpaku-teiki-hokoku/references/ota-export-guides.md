# OTA予約データ エクスポートガイド

このガイドは、各OTA（予約サイト）・PMS・メールから予約データを**ファイルとして取り出し、作業フォルダに入れる**ための手順集です。取り込んだファイルは、スキルとの対話を通じて正規化CSV（`normalized_reservations.csv`）に変換され、集計エンジンが定期報告の数値を計算します。

**信頼度の表記について（正直な記載方針）**

- ✅ **確認済み** — 実際のエクスポートファイルまたは公式文書で直接確認した情報
- ⚠️ **未確認** — 公開情報では確認できなかった情報。記載どおりでない可能性があります。未確認の形式に出会った場合は、必ず本ガイド末尾の「未知フォーマットのプロトコル」（§7）に従い、推測で処理を進めません。

---

## 1. Airbnb（ホスト向け予約一覧CSV）— ✅ スキーマ確認済み

Airbnbのホスト向け予約CSVは、このスキルで**唯一スキーマを実物で検証済み**の入力形式です（英語ロケールの実エクスポート、2025年12月〜2026年6月時点で不変）。

### 1.1 列構成（13列・順序固定）

| 列 | 英語ヘッダー（verbatim） | 日本語ヘッダー | 内容 |
|---|---|---|---|
| A | `Confirmation code` | 確認コード ✅ | 予約コード。観測例はすべて `HM` + 英数字8桁（保証ではない） |
| B | `Status` | ステータス ✅ | 表示用文字列。**きれいな enum ではない**（下記参照） |
| C | `Guest name` | ゲスト名 ✅ | 予約者（代表者）の表示名のみ。**同行者は行として展開されない**（1予約=1行） |
| D | `Contact` | 連絡先 ⚠️ | 電話番号文字列。空欄のことがある。メールアドレス列は存在しない |
| E | `# of adults` | 大人の人数 ⚠️ | 整数 |
| F | `# of children` | 子どもの人数 ⚠️ | 整数 |
| G | `# of infants` | 乳幼児の人数 ⚠️ | 整数。**定期報告の宿泊者数には乳幼児も含めて数えます**（公式ルールに年齢区分なし） |
| H | `Start date` | 開始日 ✅ | チェックイン日 |
| I | `End date` | 終了日 ✅ | チェックアウト日 |
| J | `# of nights` | 宿泊日数 ⚠️ | 泊数（整数） |
| K | `Booked` | 予約済み ✅ | 予約成立日 |
| L | `Listing` | リスティング ✅ | リスティング**名**のみ。**リスティングID列は存在しない**ため、リスティング名→届出住宅の対応付けは設定ファイル（`minpaku-config.yaml`）で管理します |
| M | `Earnings` | 収入 ✅ | 通貨記号・桁区切り付きの文字列（例 `¥45,000`、キャンセル行は `¥ 0`） |

日本語ヘッダーの ⚠️ 5列は、2023年10月時点の実CSVスクリーンショットで**先頭部分のみ**確認できたものの推定完全形です（例:「大人の人…」→ 大人の人数）。このため、スキルの解析は**列位置 + ヘッダーのあいまい一致**で行い、ヘッダー文字列の完全一致には依存しません。日本語ロケールのヘッダーが2026年現在も同一かどうかも未確認です（英語ヘッダーの2025→2026不変が傍証）。

### 1.2 日付形式の注意（✅ 確認済み・ロケール間で不統一）

| 列 | 英語ロケール | 日本語ロケール |
|---|---|---|
| `Start date` / `End date` | `M/D/YYYY`（ゼロ埋めなし。例 `9/9/2026`） | `YYYY/M/D`（例 `2023/7/30`）⚠️ 2023年時点の確認 |
| `Booked` | `YYYY-MM-DD` | `YYYY-MM-DD`（**両ロケール共通**） |

同一ファイル内で日付形式が混在する、という点に注意してください。スキルは変換時にすべて `YYYY-MM-DD` に正規化します。

### 1.3 Status 列の実態（✅ 観測値／⚠️ 全量は未確認）

観測された値: `Confirmed` / `Currently hosting` / `Past guest` / `Review guest` / `Review guest - Expires soon` / `Canceled by guest`。日本語ロケールでは 過去のゲスト / 現在ホスティング中 / ゲストのレビュー待ち系 / ゲストによりキャンセル済み系。

- キャンセル判定は「`Cancel` / `キャンセル` を含むか」の部分一致で行います。
- ホスト起因・Airbnb起因のキャンセルや問い合わせ段階の文字列は**未観測**です。**未知のステータス値は黙って捨てず、必ず一覧にして確認を求めます。**

### 1.4 ファイル仕様（✅ 確認済み）

UTF-8（BOMなし）、改行LF、全フィールドがダブルクォート囲み。ダウンロードファイル名は `reservations.csv`（同名重複時はブラウザが `reservations (1).csv` 等に連番化）。国籍・国・住所の列は**一切ありません**（§6の国籍ソースが別途必要）。

### 1.5 エクスポート手順（クリックパス）

1. ホストアカウントでログインし、予約一覧ページを開く: `airbnb.com/hosting/reservations`（日本語版は `airbnb.jp/hosting/reservations/`）。
2. タブ（今後の旅行 / 完了した予約 / キャンセル済みの予約 / すべて）と「フィルター」で**報告対象期間**（対象2か月に1泊でもかかる予約が含まれる範囲）に絞り込む。
3. 「エクスポート」→「CSVファイルをダウンロード」を選ぶ。
4. ダウンロードされたCSVを作業フォルダ（スキルが案内する場所）に保存する。

⚠️ 上記の画面構成は2023年10月の実画面と2025年のコミュニティ報告に基づきます。UI変更により、現在は「今日」タブ →「すべての予約を見る」経由に移動している可能性があります。ボタンが見つからない場合は、予約一覧画面内の「エクスポート」に相当するメニューを探してください。

### 1.6 ⚠️ 重要:「約40行の罠」（ページ単位エクスポート）

「CSVファイルをダウンロード」は、**フィルター結果の全件ではなく、画面に表示中の1ページ分（約40行）しか出力しません**。「すべて」を選んでいても同じです（2026年6月時点で確認された挙動。40という数字自体はAirbnb非公表で、全アカウント共通かは未確認）。

対策:

- 予約が約40件を超える場合は、**ページを送りながら1ページごとにダウンロード**し、複数のCSVをすべて作業フォルダに入れてください（結合と確認コードによる重複排除はスキルが行います）。
- または日付フィルターで期間を細かく区切り、各区間が1ページに収まるようにしてから出力してください。
- **検算**: ダウンロードした行数の合計が、画面上の該当予約件数と一致するか必ず確認してください。足りない場合は取りこぼしています。

### 1.7 共同ホスト（コホスト）の場合（⚠️ エクスポート可否は未確認）

Airbnbのコホスト権限は「フルアクセス」「カレンダーとメッセージ」「カレンダーのみ」の3段階です（公式ヘルプ記事1534）。フルアクセスは取引履歴・収益ダッシュボードを閲覧できますが、**予約一覧のCSVエクスポートボタンをコホストが使えるかどうかは、どこにも公式文書がありません**。

手順: まず自分のコホストアカウントでエクスポートを**試してください**。ボタンが出ない・機能しない場合の代替は次の2つです。

1. オーナー（アカウント所有者）本人にエクスポートを依頼し、CSVファイルを受け取って作業フォルダに入れる。
2. Gmailの予約確定メールを使うフォールバック（§6）。

## 2. Booking.com（エクストラネット）— ⚠️ 列構成は未確認

**確認できていること:**

- エクストラネット（管理画面）から、日付範囲を指定して予約一覧をダウンロードできる（XLS→CSV系）。財務面では月次の予約明細（reservation statements）もダウンロードできる。
- チャネルマネージャー向け接続API（B.XML）には顧客の `countrycode` があるが、これは**居住国であって国籍ではない**。

**確認できていないこと（正直に）:**

- ホスト向けダウンロードファイルの**正確な列構成**（公式パートナーヘルプは外部からの取得を拒否するため検証不能でした）。
- ゲストの国・国籍に類する列が含まれるか。仮に含まれていても居住国≠国籍である点に注意。

**手順（マッピング確認フロー）:**

1. エクストラネットの予約セクションから、報告対象期間の予約一覧をダウンロードする。
2. ファイルを作業フォルダに入れる。
3. スキルが検出した列の一覧と正規化スキーマへの対応案を提示するので、**内容を確認してから**処理を進める（§7のプロトコル）。

なお、Booking.comの予約確認**メール**の本文フィールドは未確認のため、Booking.comについてはメールではなく**エクストラネットのダウンロードを優先**してください（§6.4）。

## 3. Rakuten Vacation Stay / Agoda / Expedia — ⚠️ いずれも未確認

- **Rakuten Vacation Stay**: 管理画面の予約一覧に「予約データ出力」ボタンがあり、出力項目を設定できるとの断片情報あり（検索スニペットのみ。列の詳細・国籍系項目の有無は**未確認**）。
- **Agoda（YCS）** / **Expedia（Partner Central）**: エクスポートの有無・内容とも公開文書からは**確認できませんでした**。

いずれも手順は共通です: 管理画面でエクスポート機能を探す → 対象期間で出力 → ファイルを作業フォルダに入れる → スキルが検出列とマッピング案を提示 → **確認してから**処理（§7）。エクスポート機能が見つからない場合は、予約一覧画面のコピー&ペーストやスクリーンショットでも構いません（その場合も同じ確認フローを通します）。

## 4. PMS・チャネルマネージャーをお使いの場合（AirHost / Beds24 / ねっぱん! など）

PMS（宿泊管理システム）やチャネルマネージャーを導入済みであれば、**そのエクスポートが最も良い入力**になり得ます。複数OTAの予約が一元化されており、チェックインフォーム由来の**国籍列を持つ場合がある**ためです。

- **AirHost**: 宿泊者データエクスポート（国籍列を含むゲスト台帳CSV）が存在します（✅ 公式ヘルプで確認。ただし有料のチェックインプラン契約とオーナー/マネージャー/管理者権限が必要）。⚠️ ダウンロード可能期間が約3か月に限られるため、**定期的にエクスポートして保存**してください。定期報告形式のワンクリック出力があるかは未確認です。
- **Beds24**: Beds24 Japan が**無料のGoogleスプレッドシート製 定期報告CSVジェネレーター**（汎用版・Airbnb専用版）を公開しています。既にBeds24をお使いなら、そちらを使うのも十分合理的な選択肢です（国籍は手入力が残る「半自動」と自認されています）。もちろんBeds24のカスタムレポート出力をこのスキルに入力しても構いません。
- **ねっぱん!**: サイトコントローラーであり、定期報告向け機能は確認できませんでした（⚠️）。予約一覧の出力機能があればそれを入力にできます。

PMSのエクスポート形式はいずれも**スキーマ未確認**のため、初回は必ず §7 の確認フローを通します。

## 5. 宿泊者名簿・チェックインフォームのデータ

OTAエクスポートには国籍が**含まれません**（Airbnbは確認済み、他社も期待できません）。国籍は次のいずれかから取り込みます。

- 宿泊者名簿（Excel / Googleスプレッドシート。Sheetsの場合は「ダウンロード → Microsoft Excel (.xlsx)」で作業フォルダへ）
- 事前チェックインフォームの回答（フォーム→スプレッドシートのエクスポート）
- パスポートの写し（外国人宿泊者で日本国内に住所を有しない方の分は、施行規則・ガイドライン〔施行要領〕上そもそも取得し名簿とともに3年保存する義務があります）

名簿をまだ整備していない場合は、スキル同梱の宿泊者名簿テンプレート（`assets/guest-registry-template.xlsx`）をご利用ください。

## 6. Gmail 予約確認メール フォールバック

CSVエクスポートが使えない場合（コホスト権限の制約、過去データの欠落など）の代替経路です。

### 6.1 Airbnbホスト向け「予約確定」メールに含まれるもの（✅ 検証済み）

実物のメール本文（2023〜2024年の保全サンプル）と2025年9月時点の解析コードに基づき、以下を確認済みです。

- 送信元: `automated@airbnb.com`
- 件名: `Reservation confirmed - {ゲスト名} arrives {月 日}`（`New booking confirmed! …` の変種あり）
- Gmail検索クエリ（2025年9月時点で有効 ⚠️ 将来変更の可能性あり）: `from:automated@airbnb.com subject:"Reservation confirmed -"`

| 項目 | 含まれるか |
|---|---|
| ゲスト名（代表者） | ✅ 件名・本文 |
| チェックイン/チェックアウト日（時刻付き） | ✅ |
| **人数内訳（大人・子ども・乳幼児）** | ✅ GUESTS ブロック（例 "1 adult, 2 children"。乳幼児がいる場合は itemize される） |
| 泊数 | ✅（"× N nights" の行から） |
| リスティング名 | ✅ |
| 確認コード（`Confirmation code`） | ✅ 専用ブロック |
| 料金・ホスト受取額 | ✅ |
| **居住国コード**（例 "US"） | ✅ ただし**国籍ではない**（弱い手がかりに過ぎない） |
| 国籍 | ❌ **含まれない** |
| ゲストの電話番号・実メールアドレス | ❌ 含まれない |

つまりメール経路で埋められるのは**宿泊日数・宿泊者数・延べ宿泊者数**の側であり、**国籍22分類は埋まりません**（§5の国籍ソースが別途必要）。

⚠️ 未確認事項:

- **日本語版**ホスト向け確認メールの正確なフィールド見出し（大人/子ども/乳幼児 の表記、確認コードブロックの体裁）は未確認です。最初の1通をスキルに読ませて、お使いの言語・テンプレートを解析できるか**必ず検証**してから一括処理に進みます。
- **キャンセル通知・予約変更（alteration）メール**の現行の件名・本文フィールドは未確認です。キャンセルや日程変更がある場合は、該当メールも一緒にフォルダに入れ、1通ずつ確認します。
- チャネルマネージャー接続中のアカウントでメール本文が簡略化されるという説は、公開情報では**確認できませんでした**。お手元のメールに受取額ブロックまで表示されているかをご確認ください。

### 6.2 メールを作業フォルダに入れる方法（OS・件数別）

データの機械可読性は `.eml`（生メール）> `.mbox` > PDF の順に高くなります。PDF化は最終手段としてください。

1. **Gmailの1通ずつダウンロード（.eml）** — 少量（〜20通程度）向け。メールを開く → 右上の「⋮（その他）」→「メッセージをダウンロード」。`.eml` ファイルがダウンロードされるので作業フォルダへ移動（✅ Google公式ヘルプで確認。パソコン版のみ）。
2. **Google Takeout でラベルごと一括（.mbox、合計10クリック程度）** — 大量の過去分向け。手順: (a) Gmailで検索 `from:automated@airbnb.com` → フィルタ作成 → ラベル（例 "Airbnb"）を付与、(b) takeout.google.com → すべて選択解除 → 「メール」のみ選択 → 「メールのすべてのデータが含まれます」→「すべてのメッセージを含める」のチェックを外し、作成したラベルだけを選ぶ → エクスポート。zip内の `.mbox` ファイルを作業フォルダへ（✅ 公式ヘルプで確認。エクスポート完了まで数分〜数日かかることがあります）。
3. **Apple Mail のドラッグ保存（Mac・一括向け）** — GmailアカウントをMacに追加後、複数メールを選択して Finder のフォルダへドラッグすると1通=1つの `.eml` として保存されます（✅ Apple公式ガイドで確認）。
4. **Thunderbird の「名前を付けて保存」（Windows・一括向け）** — Thunderbird（無料）にGmailを設定 → 検索/ラベル表示で複数選択 → 右クリック → 名前を付けて保存 → フォルダを指定すると1通=1 `.eml` で保存されます（⚠️ 手順はコミュニティ情報ベース。アドオン不要とされる）。

### 6.3 メール経路の限界（再掲・重要）

- 国籍はどのOTAメールにも含まれません。
- 確認メールだけではキャンセル・日程変更が反映されません。キャンセル/変更メールも必ず一緒に入れるか、最終的な予約一覧と突き合わせてください（確認コードで重複・矛盾を検出します）。

### 6.4 Booking.com のメールについて

新規予約時にメール通知が来ること自体は確認済みですが、**本文のフィールド構成は未確認**です。Booking.comはメールではなくエクストラネットのダウンロード（§2）を第一選択にしてください。

## 7. 未知フォーマットに出会ったときのプロトコル（全ソース共通）

Airbnbの予約CSV以外のスキーマは、本ガイド執筆時点で**実物検証ができていません**。そこで、初めて見る形式のファイルは必ず次の手順で処理します。

1. **検出結果の提示**: スキルがファイルのヘッダー行と先頭数行を読み、「検出した列」と「正規化スキーマへの対応案」を表にして提示します。正規化スキーマ（列名は英語のまま固定）:

   ```
   property_key,listing_name,source,confirmation_code,guest_name,checkin,checkout,adults,children,infants,status,nationalities
   ```

2. **1回の確認質問**: 「この対応付けで正しいですか？」に答えてから処理が進みます。対応先が見つからない列、人数内訳が取れない場合、日付形式が曖昧な場合は、その旨を**明示的に**列挙します。推測のまま黙って進めることはありません。
3. **不明値の扱い**: 解釈できないステータス値・欠損は集計から黙って落とさず、一覧にして判断を仰ぎます。
4. **（任意）ヘッダー共有のお願い**: 差し支えなければ、そのファイルの**ヘッダー行のみ**（ゲスト情報の行は絶対に含めないでください）を、本リポジトリのGitHub Issueテンプレート「OTAエクスポートのヘッダー共有」から共有いただけると、次のバージョンで対応形式を増やせます。
