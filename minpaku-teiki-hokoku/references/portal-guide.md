# 民泊制度運営システム 提出ガイド

定期報告の提出先である **民泊制度運営システム**（ https://www.minpaku.mlit.go.jp/ ）の使い方と、既知の落とし穴。ガイドラインは「定期報告は、民泊制度運営システムを利用して行うことを原則とする」と定めている（一部自治体は紙・メール等も受け付ける。京都市は独自様式 — `references/kyoto.md` 参照）。

主な出典: 住宅宿泊事業者向け操作手順書 v1.7（2023-03-28） https://www.mlit.go.jp/kankocho/minpaku/business/system/content/001596131.pdf （以下「操作手順書」）。最終確認日 2026-07-02。**システムの画面は手順書のスクリーンショット（2017年頃）から変わっている可能性がある** — 画面の文言が一致しないときは慌てず、同じ意味のボタンを探すこと。

提出方法は2つだけ: **Web画面での直接入力** と **CSV一括アップロード**。API は存在しない。

---

## 1. ログインとアカウント

- システムは **独自のID・パスワード**（メールアドレスベース）。GビズIDではない。ログイン画面に「サインアップ」「パスワードを忘れた場合」のリンクがある。
- 新規登録（操作手順書 第3章第1節）: ログイン画面のサインアップ → 事業者区分・姓名・メールアドレス入力 → CAPTCHA → `hqt-minpakusystem@ml.mlit.go.jp` から仮登録メール → パスワード設定。
- **IDは届出者（事業者）本人のメールアドレスで作る。** 行政書士や管理会社が入力作業を手伝う場合でも、観光庁の公式指示により **届出者本人に発行されたIDを使う**（代理人のメールアドレスをIDにはできない）。出典: 「代理人による民泊制度運営システムの利用について」 https://www.mlit.go.jp/kankocho/content/001225372.pdf
- 管理業者ロールのアカウントには定期報告機能が **ない**（管理業者向け操作手順書に定期報告の項目が存在しない）。定期報告は事業者ロールのアカウントでのみ可能。
- **ID・パスワードが分からなくなった場合**: ログイン画面の「パスワードを忘れた場合」を試し、解決しなければ **民泊制度コールセンター 0570-041-389** に問い合わせる。
- 未確認情報: ログインユーザー名は「登録メールアドレスの末尾に jj を付けたもの」だとする2023年のホスト系ブログ情報がある（Salesforce系システムのユーザー名一意制約と整合的だが、公式文書にはなく現在も同じかは未確認）。ログインできないときに試す価値はある。

## 2. 方法1: Web画面での直接入力（『事業実績報告登録』画面）

エンジンが生成する **click-guide**（`click-guide_<期間>.md`、例: `click-guide_2026-06_2026-07.md`）はこの画面の項目順をそのまま写している。画面を開いたら、ガイドの数字を上から順に転記するだけでよい。

### 2.1 画面の開き方（3通り）

1. 『ホーム』→「事業実績一覧」→「新規事業実績」ボタン
2. 『ホーム』→「届出一覧（受理済）」→ 対象の届出にチェック →「事業実績」ボタン
3. 『ホーム』→「届出一覧（受理済）」→ 届出番号のリンク → 『事業届出詳細』下部の「実績登録」ボタン

方法2・3では **届出番号が自動入力される**（打ち間違い防止のため方法2か3を推奨）。

### 2.2 入力項目（画面上の順序どおり）

| # | 項目 | 入力方法 |
|---|---|---|
| 1 | 届出番号 | 「第○○…号」形式。方法2・3なら自動入力 |
| 2 | 報告期間 | プルダウンから選択 — **下記の2つの罠に注意** |
| 3 | 宿泊日選択 | 表示された2か月分のカレンダーで **宿泊があった日を1日ずつクリック**（クリックで色が変わり、上部の宿泊日数カウントが増える。取り消しは再クリック） |
| 4 | 宿泊者数国籍別 | 22個の数値欄（日本〜その他）に人数を入力 |
| 5 | 宿泊日数 | 自動計算（カレンダーのクリック数）— **手元の数字と一致するか照合する** |
| 6 | 宿泊者数 | 自動計算（国籍別の合計）— **手元の数字と一致するか照合する** |
| 7 | 延べ人数 | **唯一の手入力の計算値。** 操作手順書（原文）:「宿泊者数国籍別と同様、手動で入力をしてください」 |

「保存」→『事業実績報告登録完了』画面 → 「実績報告登録完了通知」メールが届く。

### 2.3 known traps（必ず伝えること）

- **報告期間プルダウンの既定値の罠**: 既定で選ばれている期間が、いま報告すべき期間と違うことが多い（ホストブログ情報・公式文書にはない）。**必ず対象期間を自分で選び直して確認する。**
- **項目リセットの罠**（公式・操作手順書 表2-6 原文）:「報告期間を変更すると、届出番号以外の全項目がリセットされます。ご注意ください」→ **入力を始める前に報告期間を確定させる。** 途中で期間を変えるとカレンダーのクリックも国籍別入力も全部消える。
- 22欄の並び順は click-guide と同一（日本・韓国・台湾・香港・中国・タイ・シンガポール・マレーシア・インドネシア・フィリピン・ベトナム・インド・英国・ドイツ・フランス・イタリア・スペイン・ロシア・米国・カナダ・オーストラリア・その他）。0人の欄は入力不要（または0）。

### 2.4 ゼロ報告

宿泊実績がない期間も同じ画面で報告する: 宿泊日は1日もクリックせず、国籍別はすべて0のまま、延べ人数に0を入れて保存（定期報告QA #3: https://www.mlit.go.jp/kankocho/content/001247862.pdf ）。

## 3. 方法2: CSV一括アップロード（『事業実績アップロード』画面）

複数物件・複数期間を一度に出せる公式ルート。1行 = 届出番号×報告期間。操作手順書の作成イメージ（図2-46）には同一届出番号の複数期間行が並んでおり、複数行アップロードが可能。

### 3.1 ファイル形式（操作手順書 表2-7）

- 1行目: 項目名（ヘッダー）、2行目以降: データ。
- 列: 届出番号, 報告期間, 宿泊日数, 宿泊者数, 宿泊延べ人数, 国籍別22列（日本〜その他）, 宿泊日。
- 操作手順書のシリアライズ:
  - 届出番号: 「第○○…号」形式（例: 第M130000000号）。正式受理済みの番号のみ。
  - 報告期間: 「20××年度××月～××月」形式（例: 2026年度06月～07月）。**年度表記**（4月起点）である点に注意。
  - 宿泊日: 「20××-××-××」形式を「;（セミコロン）」区切り。
  - 数値列は数字のみ。
- **注意（公式文書内の誤記）**: 表2-7の国籍列の列挙は「香港」を落としているが、同手順書のスクリーンショット（図2-46）と民泊制度ポータルの22区分列挙は香港を含む。**香港列を含む22列が正**。
- 文字コード・改行コード: 公式手順書に記載がない。コミュニティの実績報告では **Shift-JIS（cp932）＋CRLF** で成功例がある（未公式・中確度）。エンジンが出力する `teiki-hokoku-upload_<期間>.csv` はこの形式（Shift-JIS＋CRLF）で生成される。

### 3.2 重要: 2つの公式マニュアルでシリアライズが食い違っている

- 操作手順書 v1.7（2023）: 届出番号「第M190000056号」/ 報告期間「2017年度04月～05月」/ 宿泊日 セミコロン区切り「20××-××-××」。
- 電子宿泊者名簿 操作説明書 v2.0（2025-09-16, https://www.mlit.go.jp/kankocho/minpaku/business/system/content/001603161.pdf ）: 届出番号「M011234567」/ 報告対象期間「2018_4-5」/ 宿泊日 カンマ区切り「2018/04/06」。
- **どちらを今のアップロード口が受け付けるかは未確認**（どちらも公式）。ヘッダー行の正式な文字列もどのマニュアルにも印字されていない。
- エンジンは既定で操作手順書形式を生成する（設定 `csv_variant: sousa-tejunsho`）。電子宿泊者名簿形式が必要なら `csv_variant: denshi-meibo` に切り替えられる。

### 3.3 必ず1行テストアップロードから始める

アップロード結果画面は **正常件数** と **行ごとのエラーメッセージ** を表示するので、少量テストが安全にできる。

1. まず **ヘッダー＋データ1行だけ** のファイルを作ってアップロードする（エンジン出力のCSVから先頭データ行だけ残して保存すればよい）。
2. 『CSVアップロード完了』画面で正常件数が1になっているか確認。**正常件数に数えられた行はその時点で報告完了になる**（取り消しではなく修正扱いになる）ので、テストに使う行も本番の正しい数字であること。
3. エラーが出たら、メッセージに従って形式を直す（形式エラーならもう一方の `csv_variant` を試す価値がある）。
4. 成功した形式で残りの行をアップロードする。

### 3.4 アップロード手順

1. 『ホーム』→「事業実績のアップロード」→『事業実績アップロード』画面。
2. 「アップロード」ボタン → ファイル選択 → 「開く」→ 元の画面で「登録」ボタン。
3. 『CSVアップロード完了』画面で正常件数とエラーメッセージを確認。エラー行は修正して再アップロード。

## 4. 提出後の閲覧・修正・期限超過

### 4.1 修正できるのは期限当日まで

操作手順書 4-4（原文）:「報告した事業実績は、規定の期間内(報告期間の翌月の15日まで)に限り修正ができます…例）4月5月の報告は6月15日までは修正可能となります」

- つまり **修正窓口は提出期限そのもので閉まる**。修正は『事業実績報告詳細』画面下部の「変更」ボタンから（修正可能な期間のみボタンが表示される）。修正しても完了通知メールは来ない。
- 例外: 同一物件について別事業者の報告期間と重複している場合は、期間内でも修正不可（「変更」ボタンが出ず、メッセージが表示される）。
- **期限を過ぎた後・例外ケースの修正は、自治体の行政担当者に連絡して手続きする**しかない。
- 注: 古い「定期報告に関するQA」には「事業者による修正や上書きができません」とあるが、これは修正機能追加（手順書v1.3, 2019年）以前の記述で **現在は古い**。操作手順書が現行の正。
- **含意**: 提出前の人間による最終確認が唯一の安全弁。エンジンの出力（`validation_<期間>.md` と各物件のレビュー用xlsx）を必ず確認してから提出すること。

### 4.2 過去の報告の閲覧 — エクスポート機能はない

- 閲覧は2通り: 『事業実績一覧』で報告期間をクリック / 『事業者届出詳細（受理済）』下部の「定期報告の内容」一覧（報告期間・宿泊日数・宿泊者数・延べ人数）から開く。
- **提出済み報告のCSV/PDFダウンロード機能はどの公式文書にも記載がない**（ログインして未記載の機能を探す検証は未実施）。過去数値との突き合わせはシステムからは取れない前提で、**自分の手元記録を残すこと** — エンジンが期間フォルダに出力するファイル一式（report_data.json、レビュー用xlsx等）を消さずに保管しておけば、それが次回以降の照合記録になる。
- 年度内の累計宿泊日数（180日カウント）は届出住宅の受理済みデータ詳細画面で確認できる。

### 4.3 リマインドと警告メール

- 報告時期が近づくとシステムからリマインドメールが届く。ただし操作手順書は「リマインドメールの有無によらず、住宅宿泊事業者の責任においてスケジュール管理を」と明記。偶数月の10日頃にカレンダーリマインダーを設定しておくのが安全。
- 年度内累計宿泊日数 120〜169日: 件名「住宅宿泊事業　人を宿泊させた日数に関する注意喚起」/ 170日以上: 件名「要注意　住宅宿泊事業　人を宿泊させた日数に関する注意喚起」。

### 4.4 期限に間に合わなかった場合

- 法的には報告義務違反（罰則は `references/legal-requirements.md` §7）。実務上の一般的な助言は「気づいた時点でできるだけ早く提出し、所管窓口に電話する」。
- 未確認: 報告期間プルダウンに過去のどこまでの期間が出るか（大幅に遅れた報告をシステムから出せるのか、自治体経由でしか出せないのか）は文書化されていない。窓口に確認すること。
- 未確認（ブログのみの情報）: 未報告だと保健所から電話があり記録に「要注意マーク」が付く、という体験談があるが公式裏付けなし。

## 5. 紙・メール等の代替ルート（自治体依存）

- 原則はシステムだが、自治体によっては独自様式の紙・Excel・FAX・メール提出を受け付ける（例: 埼玉県 https://www.pref.saitama.lg.jp/a0806/minpaku/jutaku-shukuhaku12.html ）。
- 京都市は条例の追加項目（苦情の状況）を含む独自様式での提出 — `references/kyoto.md` に詳細。
- 自分の自治体の窓口・条例は必ずここで確認: https://www.mlit.go.jp/kankocho/minpaku/municipality.html

## 6. 未確認事項まとめ（fail-loud）

| 事項 | 状態 | 回避策 |
|---|---|---|

| CSVヘッダー行の正式文字列 | どのマニュアルにも未印字 | 同上（行ごとのエラーメッセージで検証できる） |
| 文字コード Shift-JIS＋CRLF | コミュニティ実績のみ | 同上 |
| ログインID = メール+「jj」 | ブログ1件のみ | ログイン不可時に試す。だめならコールセンター 0570-041-389 |
| 報告期間プルダウンの遡及範囲 | 未文書化 | 大幅遅延時は窓口に電話 |
| 提出済み報告のエクスポート | 記載なし（ログイン検証未実施） | 手元記録（エンジン出力）を保管 |
| 現在の画面UI（手順書のスクショは2017年頃） | 変わっている可能性 | 画面文言が違っても同義のボタンを探す |
